# 授信报告数据核验设计方案

## 1. 背景

客户的真实顾虑不是“AI 能不能生成报告”，而是：

- 报告虽然由 AI 生成，但审批责任仍在人工
- 人工必须核对报告中的关键数据是否正确
- 如果人工还要全文重算一遍，AI 生成报告的价值会被削弱

因此，后续 demo 需要补一条新能力：

**不是继续做“AI 审核意见”，而是做“数据核验 / 数据可追溯核验”。**

这个能力的目标不是替人工担责，而是：

- 帮人工快速看到报告中的数字来自哪里
- 区分“直接取数”和“计算得出”
- 给出必要的口径说明和公式说明
- 将“全文重算”缩减为“按来源链路抽查”

## 2. 为什么优先选授信报告，不选代偿补偿报告

代偿补偿报告的数据来源链路过于复杂：

- 涉及多个材料和不同类型文件
- 生成中间过程较多
- 当前缓存结果里已经不容易反推出原始来源

授信报告更适合做第一版数据核验 demo：

- 核心数据主要集中在一份 Excel 统计表
- 生成前已有较完整的结构化数据层
- 很多报告数字本身就是统计表里的汇总值、分组值或行级值
- 更容易讲清“直接取数 / 汇总得出 / 简单公式”

因此：

**第一版数据核验只做授信报告。**

## 3. 已讨论并否决的方案

### 3.1 事后从 AI 生成的报告中抽所有数字，再倒推来源

这个方案理论上可以做数字抽取，但不适合作为主方案。

问题在于：

- 找到“报告中的所有数字”不难
- 但很难准确判断每个数字对应什么业务含义
- 更难准确反推出它来自哪个输入字段或哪个计算过程
- 事后倒推很容易出现多个候选来源、口径误配、误把模型润色结果当成真实来源

结论：

**不能把“从最终报告倒推来源”作为主线方案。**

### 3.2 要求当前版本立即做到“报告中所有数字 100% 自动精确溯源”

当前生成链路并没有完整保留“每个数字的来源关系”。

如果在没有来源留痕的情况下强行要求全量精确溯源，会导致：

- 工程成本过高
- 模型只能猜
- 最终可信度不够

结论：

**第一版不能承诺“所有数字都自动精确溯源”。**

## 4. 当前可行主方案

### 4.1 方案核心

不要在“报告生成之后”反推，而是在“报告生成当下”让模型同时输出两份内容：

1. 正式报告正文
2. 数据引用与来源说明（data trace）

也就是采用：

**双输出方案**

- 通道 A：`report_text`
- 通道 B：`data_trace`

这样做的好处：

- 模型生成报告时，仍然持有输入数据的完整上下文
- 可以在同一次调用中同步说明“哪些数字用了哪些输入”
- 数据说明不会污染正式报告正文

## 5. 报告污染问题的解决方案

这是本方案的关键。

### 5.1 问题

如果让模型把“来源说明、公式说明、口径解释”直接写进正式报告正文，会污染报告：

- 报告不再像正式公文
- 内容会夹杂技术解释
- 不适合直接成文和提交

### 5.2 解决方式

**严格采用双输出，不允许把 trace 信息写进 report_text。**

输出结构必须拆成：

```json
{
  "report_text": "正式授信报告正文",
  "data_trace": [
    {
      "number_text": "451.3亿元",
      "semantic_label": "拟授信总额",
      "source_type": "direct",
      "source_ids": ["stats.granted_total"]
    }
  ]
}
```

约束：

- `report_text` 只保留正式公文内容
- `data_trace` 单独保存数字来源和计算关系
- UI 中“授信报告页面”只展示 `report_text`
- UI 中“数据核验页面”只展示 `data_trace`

这样可以完全避免正式报告被污染。

## 6. 为什么不能只靠模型自由描述“来源在哪”

如果只是把一堆基础数据发给模型，让模型自由声称“这个数字来自哪里”，风险很大：

- 模型可能说错来源
- 模型可能把推断结果描述成真实输入
- 模型可能生成不存在的字段名或口径名称

因此：

**必须先给输入数据做稳定编号（source id），再要求模型只引用这些 id。**

## 7. 输入数据的改造要求

### 7.1 当前问题

目前授信报告生成时，模型看到的是结构化数据，但这些数据没有统一、稳定、可引用的 source id。

### 7.2 改造原则

生成授信报告时，不再只传“值”，而要传：

- `id`
- `label`
- `value`
- `unit`
- `scope`

例如：

```json
{
  "stats": [
    { "id": "stats.institution_count", "label": "合作机构数量", "value": 41, "unit": "家" },
    { "id": "stats.application_total", "label": "申请总额", "value": 489.3, "unit": "亿元" },
    { "id": "stats.granted_total", "label": "拟授信总额", "value": 451.3, "unit": "亿元" }
  ]
}
```

以及分组字段：

```json
{
  "groups": [
    {
      "id": "groups.policy_target.total_credit",
      "label": "按政策目标值配置合计",
      "value": 233.1,
      "unit": "亿元"
    }
  ]
}
```

再到机构层：

```json
{
  "institutions": [
    {
      "id": "institutions.xa_small_micro.total_credit",
      "label": "西安小微企业融资担保有限公司拟授信额度",
      "value": 12.0,
      "unit": "亿元"
    }
  ]
}
```

## 8. 第一版建议覆盖的数字范围

虽然长期目标可以是“报告中的所有数字”，但第一版建议优先覆盖授信报告里最有业务价值、且最容易建立 trace 的数字。

建议至少覆盖三层：

### 8.1 统计总览类

- 合作机构数量
- 申请总额
- 拟授信总额
- 银担分险额度
- 非银担分险额度
- 国企担保贷款额度

### 8.2 分组汇总类

- 按申请额度配置合计
- 按政策目标值配置合计
- 按测算值配置合计
- 新机构按政策目标值配置合计

### 8.3 行级机构类

- 单家机构拟授信额度
- 单家机构非银担额度
- 单家机构银担额度
- 单家机构国企担保额度

说明：

- 第一版可以优先覆盖这类“已有结构化字段”的数字
- 报告中其他自然语言里出现的辅助数字，可后续再补

## 9. data_trace 建议结构

建议在生成阶段让模型输出如下结构：

```ts
type CreditReportTraceItem = {
  id: string;
  number_text: string;
  semantic_label: string;
  source_type: 'direct' | 'derived';
  source_ids: string[];
  formula?: string;
  note?: string;
};
```

说明：

- `id`: trace 项自身 id
- `number_text`: 报告中最终出现的数字文本，例如 `451.3亿元`
- `semantic_label`: 该数字对应的业务含义，例如 `拟授信总额`
- `source_type`:
  - `direct`: 直接来自输入数据
  - `derived`: 由输入数据汇总或计算得到
- `source_ids`: 只能引用输入里已定义的 id
- `formula`: 仅在 `derived` 时填写
- `note`: 可选，用于补充口径说明

## 10. 生成接口的目标输出结构

建议最终输出 schema 改为：

```ts
type GeneratedCreditReportWithTrace = {
  report_text: string;
  data_trace: CreditReportTraceItem[];
  generated_at: string;
};
```

## 11. 提示词约束建议

生成授信报告时，system / user prompt 需要新增约束：

### 11.1 对 report_text 的要求

- 只输出正式报告正文
- 不允许在正文中写来源、公式、注释、括号解释
- 不允许在正文中输出 trace id

### 11.2 对 data_trace 的要求

- 必须记录正文中使用到的关键数字
- 每个数字必须标明 `source_type`
- 每个数字必须只引用输入里存在的 `source_ids`
- 若为 `derived`，应填写公式或汇总口径
- 不允许编造不存在的 source id

### 11.3 对模型行为的额外约束

- 不允许把“模型推断”伪装成“输入中已有数据”
- 不允许编造 Excel 单元格、表名或字段名
- 如果某个数字无法明确绑定到输入字段，宁可不进入 `data_trace`，也不要伪造来源

## 12. 对“所有数字都需要有”的理解

本轮讨论中，已明确：

- 不能依赖“从最终报告全文抽数字，再倒推来源”作为主线
- 更合理的做法是“在生成时同步产出数字 trace”

因此，“所有数字都需要有”的合理落地方式应理解为：

- 尽可能让模型在生成时同步记录它真正使用到的数字
- 对于进入正式报告正文的关键数字，优先要求进入 `data_trace`
- 之后再逐步扩大覆盖面，而不是第一版就强行承诺全文 100% 精确溯源

## 13. UI 建议

左侧材料列表保持不变。

选中授信报告后，提供两个模式：

- `AI 复核`
- `数据核验`

其中 `数据核验` 模式下：

### 中间区域

- 按 `data_trace` 列出报告中的数字条目
- 点击某个数字后显示：
  - 数值
  - 业务含义
  - 直接取数 / 计算得出
  - 来源 source id
  - 公式 / 汇总口径

### 右侧区域

- 系统预置核数要点
- 待人工确认项

## 14. 对第一版工程实现的建议

### 14.1 先改数据输入

在授信报告生成时，将现有 `buildPromptPayload` 改造成带 source id 的输入结构。

### 14.2 再改模型输出 schema

让模型同时输出：

- `report_text`
- `data_trace`

### 14.3 最后再加 UI

不要先做 UI，再回头补数据。

正确顺序：

1. 输入结构加 source id
2. 输出结构加 data_trace
3. 缓存 generated-report 时一并写入 trace
4. 页面增加“数据核验”模式

## 15. 当前结论

本轮讨论后的结论是：

1. 授信报告适合作为数据核验第一版 demo
2. 不走“事后从报告倒推来源”的主方案
3. 改为“生成报告时双输出：正文 + data_trace”
4. 为避免污染正式报告，trace 只保存在独立结构中，不写入正文
5. 输入给模型的数据必须带稳定 `source id`
6. 第一版优先覆盖授信报告中最核心、最结构化的一批数字
