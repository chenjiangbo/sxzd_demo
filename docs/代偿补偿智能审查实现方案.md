# 代偿补偿智能审查实现方案

## 目标
这版不做通用 Agent 平台，只做一条可演示、可落地的单案件流水线。核心目标是把“宝鸡三家村餐饮管理有限公司”这笔代偿补偿案件，从资料包进入开始，一路做到：

1. 案件进入与基础信息归集
2. 材料清单匹配
3. 一致性核验与规则校验
4. 文书生成
5. 人工复核与提交 OA

最终输出必须包含：

1. 材料清单状态
2. 一致性与规则校验结果
3. 风险提示与待人工确认项
4. 三份文书草稿
5. 可提交 OA 的结果

## 范围边界
这版先不做：

1. 通用聊天首页
2. 多 Agent 协同
3. 通用知识库
4. 通用 Skill 平台
5. 复杂权限体系
6. 自动写回多个外部系统

必须坚持的边界：

1. 算数、日期、编号、比例等确定性规则由程序完成
2. 文件分类、字段抽取、风险解释、文书成文由 LLM 完成
3. 最终决定由人工完成

## 输入材料划分
### A 类：真实申报输入材料
真正进入审查流水线：

1. `1 代偿补偿函-三家村.pdf`
2. `2 代偿审批文件.pdf`
3. `3 营业执照及申请书.pdf`
4. `3 创业担保证明材料.pdf`
5. `7 借款用途举证材料 2021及2022合同相关资料.pdf`
6. `11.2 银行流水.pdf`
7. 资料包中实际存在的借款合同、保证合同、委托担保合同等文件

### B 类：结构化辅助输入
作为系统侧辅助数据：

1. `tmp备案数据.xlsx`
2. `tmp解保台账数据.xlsx`

### C 类：对标结果文书
不要作为审查输入，只作为输出参考样例：

1. `宝鸡三家村餐饮管理有限公司项目代偿补偿合规性自查工作底稿.pdf`
2. `宝鸡三家村餐饮管理有限公司项目再担保代偿补偿审批表.pdf`
3. `担保业务审批单-钟颖-2026-03-19.pdf`

原因：如果把 C 类一起喂给系统，等于“看着答案做题”，会削弱演示说服力。

## blackwhite 调用约束
### 使用方式
本项目调用 LLM 必须走本机 blackwhite 网关，不要直连模型厂商。

当前可用基础信息：

1. Gateway: `http://127.0.0.1:8080`
2. `.env.example` 中已有：
   `BLACKWHITH_APPID`
   `BLACKWHITH_KEY`
3. 当前文档和已验通的模型可直接使用：
   `google/gemini-2.5-flash-lite`

### 推荐封装
前端或服务层统一封装一个 `blackwhiteClient`：

1. 统一读环境变量
2. 统一设置 `X-App-Id`
3. 统一设置 `Authorization: Bearer ...`
4. 统一超时、错误码、日志
5. 统一 JSON 输出校验

### 最小调用样例
```bash
curl http://127.0.0.1:8080/proxy/openrouter/chat/completions \
  -H 'X-App-Id: SXZD_DEMO' \
  -H 'Authorization: Bearer <BLACKWHITH_KEY>' \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "google/gemini-2.5-flash-lite",
    "stream": false,
    "messages": [
      {"role": "user", "content": "Reply with exactly: ok"}
    ]
  }'
```

### 接入要求
1. 不要在浏览器里直接保存平台管理密钥之外的其他上游密钥
2. 所有 LLM 结果都要带原始请求 ID、prompt ID、模型名、耗时和 token 信息
3. 核心步骤不允许静默 fallback，失败就明确报错并阻断下一步

## 总体架构
推荐拆成 4 类能力：

1. 文件解析层
2. 材料匹配层
3. 规则校验层
4. 文书生成层

推荐的数据流：

1. 案件对象 `case`
2. 文件对象 `document`
3. 材料项对象 `material_item`
4. 字段抽取对象 `field_extraction`
5. 规则结果对象 `rule_result`
6. 风险提示对象 `risk_item`
7. 文书草稿对象 `draft_document`

## 步骤 2：材料清单匹配
### 页面原则
这一步的主界面就是“材料清单”。

不要再单独做：

1. 文件引用入口
2. 缺口说明

每一行材料项本身就是入口，点击后展开：

1. 匹配到的文件
2. 关键片段
3. 提取到的字段
4. AI 判断原因
5. 如果缺失，建议补什么

### 建议材料清单
先做这 10 项：

1. 代偿补偿申请函
2. 代偿审批文件
3. 借款合同
4. 保证合同
5. 委托担保合同
6. 债务人基础资料
7. 贷款用途举证材料
8. 代偿证明 / 代偿通知
9. 代偿支付凭证 / 银行流水
10. 接收补偿资金账户资料

### 状态
只保留 4 个：

1. 已匹配
2. 待确认
3. 缺失
4. 不适用

### 本案要特别支持的判断
1. 一个文件匹配多个材料项
2. 并件识别：营业执照 / 申请书 / 创业担保证明
3. 缺失项建议补件
4. 不适用项要给理由

### 这一阶段 LLM 的职责
只做文件分类与材料匹配，不做规则结论。

### 提示词：文件分类
```text
id=step2_file_classify

你是代偿补偿案件文件分类助手。

给你一个文件的标题、OCR文本前几段、页面关键词。
请判断该文件最可能属于以下哪一类：

1. 代偿补偿申请函
2. 代偿审批文件
3. 借款合同
4. 保证合同
5. 委托担保合同
6. 债务人基础资料
7. 贷款用途举证材料
8. 代偿证明或代偿通知
9. 代偿支付凭证或银行流水
10. 账户资料
11. 其他

输出 JSON：
- category
- confidence
- reason
- multiple_categories_possible
- alternative_categories

不要编造文件中不存在的信息。
```

### 提示词：材料匹配
```text
id=step2_material_match

你是代偿补偿案件材料清单匹配助手。

给你：
1. 本案材料清单
2. 每个文件的分类结果
3. 每个文件的标题和摘要

请为每个材料项判断状态：
- matched
- pending_confirm
- missing
- not_applicable

要求：
- 如果有高置信度文件支撑，则为 matched
- 如果有候选文件但不够稳，则为 pending_confirm
- 如果本案应提供但未发现，则为 missing
- 如果按本案情形不适用，则为 not_applicable
- 一个文件可以匹配多个材料项

输出 JSON 数组，每项包括：
- material_name
- status
- matched_files
- ai_reason
- manual_attention
```

## 步骤 3：一致性核验与规则校验
### 建议抽取字段
只抽这 14 个字段：

1. 债务人名称
2. 统一社会信用代码
3. 当前法定代表人
4. 原法定代表人
5. 债权人名称
6. 主债权金额
7. 借款合同号
8. 保证合同号
9. 委保合同号
10. 主债权到期日
11. 代偿日期
12. 未清偿本金
13. 省级再担责任比例
14. 申请补偿金额

### 建议规则
先做这 10 条：

1. 债务人名称一致
2. 统一社会信用代码一致
3. 借款合同号一致
4. 保证合同号一致
5. 委保合同号一致
6. 主债权金额一致
7. 申报时效是否在 120 日内
8. 补偿金额是否等于未清偿本金 × 40%
9. 首次备案记录与展期记录是否可关联
10. 法定代表人变更是否需要人工确认

### 本案重点展示
1. 申报时效：49 天
2. 补偿金额：`1,929,419.58 × 40% = 771,767.83`
3. 法代变更：给“待人工确认”，不是直接报错
4. 银行名称标准化：两种写法归一

### 程序做什么
1. 日期计算
2. 金额计算
3. 编号精确比对
4. 标准化归一
5. 规则结果状态判定

### LLM 做什么
1. 从非结构化材料中抽字段
2. 把程序结果解释成人话
3. 生成风险提示

### 提示词：字段抽取
```text
id=step3_extract_fields

你是代偿补偿案件字段抽取助手。

请从给定文件内容中抽取以下字段：
- 债务人名称
- 统一社会信用代码
- 当前法定代表人
- 原法定代表人
- 债权人名称
- 主债权金额
- 借款合同号
- 保证合同号
- 委托担保合同号
- 主债权到期日
- 代偿日期
- 未清偿本金
- 省级再担责任比例
- 申请补偿金额

输出 JSON：
- field_name
- value
- confidence
- source_excerpt

找不到就返回 null，不要编造。
```

### 提示词：规则解释
```text
id=step3_rule_explain

你是代偿补偿案件规则解释助手。

给你：
1. 规则名称
2. 程序计算结果
3. 相关字段
4. 证据摘要

请输出：
- conclusion：通过 / 待确认 / 不通过
- plain_explanation：用业务人员能理解的话解释
- risk_level：低 / 中 / 高
- manual_attention：需要人工特别看什么

不要重新计算程序结果。
不要编造制度条款。
输出 JSON。
```

### 提示词：风险提示
```text
id=step3_risk_summary

你是代偿补偿案件风险提示助手。

请根据案件字段、规则结果和材料状态，输出最值得业务人员关注的风险项。

要求：
- 只输出需要人工关注的事项
- 不要重复普通通过项
- 每条风险包含：
  1. 风险点
  2. 为什么提示
  3. 建议看什么材料
  4. 是否影响继续生成文书

输出 3 到 6 条，JSON 格式。
```

## 步骤 4：文书生成
### 生成目标
1. 合规性自查工作底稿
2. 再担保代偿补偿审批表
3. OA 审批单草稿

### 不要这样做
不要把整包材料直接丢给模型说“帮我写审批表”。

### 正确做法
1. 先把案件和规则结果整理成结构化 JSON
2. 再按不同文书模板分别生成
3. 缺失信息标记 `【待补充】`
4. 人工需要判断的段落标记 `【人工复核】`

### 文书输入
1. 案件基础信息
2. 材料清单结果
3. 字段抽取结果
4. 规则判定结果
5. 风险提示

### 提示词：文书生成
```text
id=step4_generate_doc

你是代偿补偿案件文书生成助手。

请根据：
1. 案件基础信息
2. 材料清单结果
3. 字段抽取结果
4. 规则判定结果
5. 风险提示
6. 当前文书模板

生成正式、克制、符合金融机构内部行文风格的文书草稿。

要求：
- 不要编造缺失信息
- 缺失信息用【待补充】标记
- 需要人工重点确认的段落用【人工复核】标记
- 结论必须与规则结果一致
- 金额必须与程序结果一致
- 行文正式、简洁、专业

输出：
- title
- body
- manual_review_points
```

建议分 3 次调用，不要一条 prompt 同时生成 3 份文书。

## 步骤 5：人工复核与提交 OA
### 页面展示
1. AI 审查结论摘要
2. 风险提示
3. 待人工确认项
4. 三份文书草稿
5. OA 流程预览

### 本案建议固定展示的待确认项
1. 法定代表人变更与反担保责任连续性
2. 首次备案与展期业务链关联
3. 贷款用途历史追溯说明是否接受
4. 债权人名称标准化归一是否接受

### 页面操作
1. 采纳并提交 OA
2. 修改后提交
3. 退回补充材料
4. 导出审查包

### 可选提示词：提交前摘要
```text
id=step5_review_summary

你是代偿补偿案件复核摘要助手。

请根据案件审查结果，生成一段提交 OA 前的简要说明。

要求：
- 50 到 120 字
- 说明本案是否建议进入 OA
- 提醒最重要的 1 到 3 个待确认事项
- 用语正式、简洁

不要编造材料中不存在的信息。
```

## 工程实现建议
### 后端能力拆分
建议拆成下面几个服务：

1. `case-ingest-service`
2. `document-parser-service`
3. `material-match-service`
4. `field-extract-service`
5. `rule-engine-service`
6. `risk-summary-service`
7. `draft-generate-service`

### 前端步骤与接口建议
1. 步骤 1 页面加载时，读取案件主卡与文件列表
2. 步骤 2 页面调用材料匹配接口，返回材料清单 JSON
3. 步骤 3 页面调用规则引擎接口，返回规则结果与风险项
4. 步骤 4 页面读取结构化结果并触发三份文书生成
5. 步骤 5 页面汇总结果并组装 OA payload

### 最重要的工程约束
1. 程序结果必须可追溯到原字段和原文件
2. LLM 输出必须 JSON 化并做 schema 校验
3. 任一步核心流程失败都要报错，不允许静默跳步
4. 所有文书结论都必须和程序计算结果一致
