# 我的建议：1 个主界面 + 2 个抽屉/弹层
## 一、主界面定位
名字就叫：

**AI 综合复核工作台**

它只做一件事：

**对已生成的报告做一次审批前智能复核，并允许人工纠偏、沉淀经验。**

# 三、主界面建议分成 5 个区块
## 1. 顶部：当前复核上下文
这一条要很清楚，解决“AI 要知道自己是什么岗、在审什么”的问题。

显示：

+ 当前岗位：**综合复核岗**
+ 当前报告类型：
    - 授信简报
    - 代偿补偿报告
    - 机构评价报告
    - 授信报告
+ 当前审核模式：**自动匹配对应提示词**
+ 当前状态：待复核 / 已出首版意见 / 已反馈重审 / 可提交 OA

这里的核心是：  
**岗位统一，提示词按报告类型切换。**

也就是说：

+ 还是 1 个岗位
+ 但内部有 4 套审核提示词

这和你刚才说的完全一致。

---

## 2. 左侧：报告与材料区
这里负责告诉 AI “它在审什么”。

内容建议包括：

+ 当前报告正文
+ 关联底表 / 制度 / 样本材料
+ 报告摘要信息
+ 一键切换其他报告

比如：

+ 如果是授信报告，就挂授信统计表 + 授信办法 + 历史报告
+ 如果是机构评价报告，就挂评价底表 + 两份样本
+ 如果是代偿补偿报告，就挂审批表 / 台账 / 底稿 / 流程资料
+ 如果是授信简报，就挂对应统计表和简报模板

这样 AI 的审核不是空对空，而是“带材料复核”。

---

## 3. 中间：AI 审核意见区
这是主舞台。

用户点“请审核”后，AI 在这里输出固定结构的审核意见：

### 审核结论
+ 可提交
+ 建议修改后提交
+ 存在明显风险

### 问题清单
+ 缺什么
+ 哪儿不一致
+ 哪段解释不够
+ 哪个数字需要核

### 风险提示
+ 完整性风险
+ 一致性风险
+ 合规/口径风险
+ 成文风险

### 修改建议
+ 补哪段
+ 改哪句
+ 核哪个数
+ 增加什么解释

### 审核依据
+ 本次加载了哪套审核口径
+ 用到了哪些材料
+ 引用了哪些制度/样本/经验

这一块一定要做成**结构化**，不要只是聊天回答。  
因为你是给审批人看的，不是给普通用户看的。

---

## 4. 右侧：本岗位审核要点 / 已沉淀经验
这个区块就是你想表达的“像 skill 一样自己总结经验”。

建议名字不要叫 skill，直接叫：

**本岗位审核要点**

里面分两层：

### 系统预置要点
也就是 4 类报告各自的基础审核提示词，对用户隐藏为“审核口径”，对 demo 来说就是你写好的 4 套提示词。

例如：

+ 授信简报重点看：数字支撑、趋势表述、口径一致
+ 代偿补偿重点看：时效、责任比例、补偿基数、证据链
+ 机构评价重点看：未达标指标、分析与结论一致性
+ 授信报告重点看：申请额度、政策目标值、测算值、分组口径

### 本次沉淀经验
用户在对话里说：

+ “以后审核机构评价要特别注意未达标指标”
+ “以后审核授信报告要重点看申请额和政策目标值关系”

系统就把它提炼成一条新经验，显示在这里。  
下次再审同类报告时，自动优先加载。

这块是 demo 的亮点之一。

---

## 5. 底部：对话反馈区
这里就是你说的聊天框，但它只是“复核工作台”的一个部件，不是全部。

用户可以说：

+ 请审核这份报告
+ 请重点关注 xxx 风险
+ 你漏看了 xxx
+ 重新审核
+ 把这条经验记住
+ 下次审核这个类型报告时优先检查 xxx

然后 AI：

+ 重新生成第二版意见
+ 更新“本岗位审核要点”

这就实现了你要的：  
**审核 → 反馈 → 再审核 → 经验沉淀**

---

# 四、这个界面的核心交互流程
## 第一步：选择报告
用户从 4 类报告里选一份进入复核。

## 第二步：系统自动识别报告类型
自动切换到对应审核提示词。

这一步对用户显示成：  
**当前模式：机构评价复核**  
或  
**当前模式：代偿补偿复核**

不用说“提示词”，对外讲“审核口径”。

## 第三步：用户点击“请审核”
AI 输出首版审核意见。

## 第四步：用户反馈
例如：

+ 重点看制度口径
+ 这条风险太泛了
+ 以后注意时效
+ 你漏看了数据一致性

## 第五步：AI 生成第二版审核意见
并把用户补充提炼成经验。

## 第六步：用户决定是否采纳
最后点：

+ 采纳并提交 OA
+ 修改后再审
+ 导出审核意见

**1 个主界面，内部按区块组织。**

## 最合适的产品表述：
**AI 综合复核岗**

+ 岗位只有 1 个
+ 报告类型有 4 种
+ 审核口径有 4 套
+ 用户可反馈纠偏
+ AI 会沉淀本岗位经验
+ 最后统一提交 OA
